# ClariFlow — AI 辅助研发工作项流转与需求澄清系统 PRD

---

## 1. 项目信息

| 字段 | 内容 |
|---|---|
| Language | 中文 |
| Programming Language | Java 17 + Spring Boot 3 |
| Project Name | `clariflow` |
| 数据库 | H2（内嵌模式） |
| 文档规范 | OpenAPI / Swagger |
| 测试要求 | 核心业务规则单元测试 + 集成测试 |

### 原始需求复述

研发团队需要一个轻量级模块，用于管理工作项、跟踪状态流转、记录需求澄清问题，并通过 AI 辅助分析。工作项包含标题、描述、类型、优先级、状态、负责人、标签、验收标准、风险等级、状态历史；澄清问题包含问题内容、严重程度、状态、回答。核心业务规则：存在未解决 HIGH 级澄清问题时，禁止进入 READY 或 IN_DEVELOPMENT 状态。

## 2. 产品定义

### 2.1 Product Goals

| # | 目标 | 衡量标准 |
|---|---|---|
| G1 | 标准化工作项状态流转，确保研发流程有序可控 | 6 种状态间流转零歧义，状态历史完整可追溯 |
| G2 | 结构化澄清问题管理，在需求阶段减少理解偏差 | 每条 HIGH 级问题必须在进入开发前解决 |
| G3 | AI 辅助分析，提升需求评估的效率与质量 | 单次分析返回摘要 + 风险 + 澄清建议三项结构化结果 |

### 2.2 User Stories

| # | 用户角色 | 需求 | 价值 |
|---|---|---|---|
| US1 | 产品经理 | 创建/编辑工作项，填写标题、描述、类型、优先级、验收标准 | 确保开发团队拿到明确、完整的需求定义 |
| US2 | 开发负责人 | 对工作项触发 AI 分析，获取摘要、风险点和澄清建议 | 快速评估需求质量，减少评审遗漏 |
| US3 | 开发人员 | 按规则推进状态流转（含退回），查看状态变更历史 | 工作进度透明，状态变更可审计 |
| US4 | 测试人员 | 对工作项提出澄清问题，标注严重程度并跟踪解决状态 | 需求模糊点系统化管理，不依赖口头沟通 |
| US5 | 项目经理 | 系统自动拦截：存在未解决 HIGH 级问题时禁止进入开发 | 关键风险不流入开发阶段，合规约束自动化 |

## 3. 技术规范

### 3.1 Requirements Pool

**P0 — 必须交付**

| ID | 需求 | 说明 |
|---|---|---|
| R1 | 工作项 CRUD | 标题、描述、类型、优先级、负责人、标签、验收标准、风险等级 |
| R2 | 状态机流转 | DRAFT→ANALYZING→READY→IN_DEVELOPMENT→TESTING→COMPLETED，支持退回（如 TESTING→IN_DEVELOPMENT），COMPLETED 可回退至 TESTING |
| R3 | 状态变更历史 | 每次流转记录 from/to 状态、操作时间和操作人 |
| R4 | 澄清问题 CRUD | 关联工作项，包含问题内容、严重程度（HIGH/MEDIUM/LOW）、状态（UNRESOLVED/RESOLVED）、回答 |
| R5 | 核心业务规则 | 存在 UNRESOLVED + HIGH 级澄清问题时，禁止将工作项状态变更为 READY 或 IN_DEVELOPMENT，返回明确错误信息 |
| R6 | H2 内嵌数据库 | 启动即用，无需外部数据库依赖 |
| R7 | OpenAPI/Swagger 文档 | 所有接口自动生成，可在线调试 |

**P1 — 应该交付**

| ID | 需求 | 说明 |
|---|---|---|
| R8 | AI 分析 — 需求摘要 | 根据描述字段生成结构化摘要（Mock） |
| R9 | AI 分析 — 风险识别 | 返回风险列表及等级（Mock） |
| R10 | AI 分析 — 澄清建议 | 返回建议的澄清问题列表（Mock） |
| R11 | 统一错误码体系 | 业务异常、校验异常、AI 异常各自有明确错误码 |
| R12 | 核心规则测试 | 状态流转逻辑 + 业务拦截规则的单元/集成测试 |

**P2 — 可延后**

| ID | 需求 | 说明 |
|---|---|---|
| R13 | 乐观锁（并发保护） | 工作项更新时基于版本号防止并发覆盖 |
| R14 | Docker Compose | 一键启动开发环境 |
| R15 | 标签筛选 & 搜索 | 按类型、优先级、状态、标签组合过滤工作项列表 |

### 3.3 状态流转图

```
         ┌──────┐      ┌───────────┐      ┌──────┐
         │ DRAFT │ ──→ │ ANALYZING │ ──→ │ READY │
         └──────┘      └───────────┘      └──┬───┘
                            ↑                │
                            │                ↓
                       可回退          ┌──────────────┐
                            │          │ IN_DEVELOPMENT│
                            │          └──────┬───────┘
                            │                 │
                            │                 ↓
                            │          ┌─────────┐
                            └───────── │ TESTING │
                                       └────┬────┘
                                            │
                                    可回退  ↓
                                       ┌───────────┐
                                       │ COMPLETED │
                                       └───────────┘

规则：
- TESTING → IN_DEVELOPMENT（退回开发）
- COMPLETED → TESTING（缺陷复现，重新测试）
- 禁止 COMPLETED 直接跳回早期状态
- 存在 UNRESOLVED + HIGH 澄清问题时，禁止进入 READY 或 IN_DEVELOPMENT
```

### 3.4 Open Questions

| # | 问题 | 建议方向 |
|---|---|---|
| Q1 | "负责人"字段是否需要关联独立用户表？MVP 可用字符串，后续扩展为 User 实体 | 先字符串，P2 扩展 |
| Q2 | 是否需要工作项之间的父子/关联关系？ | 当前不做，如需要可在 P2 追加 |
| Q3 | AI 分析触发方式：手动触发 or 状态变更自动触发？ | 建议手动触发（MVP 更可控） |
| Q4 | H2 是否需要持久化到文件？内存模式重启数据丢失 | 建议 `jdbc:h2:file:./data/clariflow` 开发演示更友好 |
| Q5 | 前端技术栈未指定，是否由架构师决定？ | 建议 Vite + React + MUI + Tailwind CSS |
| Q6 | 是否需要权限控制？（如仅负责人可流转状态） | MVP 不实现，P2 考虑 |

## 4. API 预览

以下为预期核心 API（具体由架构师在技术设计中细化）：

| 方法 | 路径 | 说明 | 优先级 |
|---|---|---|---|
| GET | `/api/work-items` | 工作项列表（支持筛选） | P0 |
| POST | `/api/work-items` | 创建工作项 | P0 |
| GET | `/api/work-items/{id}` | 工作项详情 | P0 |
| PUT | `/api/work-items/{id}` | 更新工作项 | P0 |
| POST | `/api/work-items/{id}/transition` | 状态流转 | P0 |
| GET | `/api/work-items/{id}/history` | 状态历史 | P0 |
| POST | `/api/work-items/{id}/clarifications` | 添加澄清问题 | P0 |
| PUT | `/api/clarifications/{id}` | 更新澄清问题（含回答） | P0 |
| POST | `/api/work-items/{id}/ai/summarize` | AI 摘要 | P1 |
| POST | `/api/work-items/{id}/ai/risks` | AI 风险识别 | P1 |
| POST | `/api/work-items/{id}/ai/suggestions` | AI 澄清建议 | P1 |
